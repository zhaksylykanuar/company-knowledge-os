# FounderOS Playbook v2

Операционный плейбук развития company-knowledge-os. Версия 2 заменяет
исходный 23-страничный плейбук QazTwin: архитектура слоёв оттуда сохранена и
значительная часть инфраструктурных слоёв уже построена, а вот операционная
модель переработана по итогам 8 месяцев реальной разработки (FOS-001…FOS-175).

Этот документ — системный контекст для чата-ПМ и для агентов. Технические
правила работы агентов живут в [`../AGENTS.md`](../AGENTS.md) и
[`../CLAUDE.md`](../CLAUDE.md) и здесь не дублируются.

## Главный принцип

```
AI drafts -> Human approves -> System executes (gated) -> Everything is logged
-> Feedback improves context
```

Агенты собирают контекст, классифицируют по смыслу, готовят черновики и
подсвечивают важное. Человек утверждает всё значимое. Исполнение живых действий
проходит только через существующие guard-границы.

## Чему нас научила версия 1

Версия 1 была устроена «слоями»: сначала ingestion, потом normalization, потом
triage, потом digest. Результат: 92K строк production-качества инфраструктуры,
1081 тест — и ни одного дня реального использования, LLM выключен, MVP-критерии
не достигнуты. Слоистый план поощряет бесконечное безопасное строительство.

Версия 2 это исправляет четырьмя правилами, которые важнее любых чеклистов.

## Четыре правила версии 2

### Правило 1 — вертикальные срезы

Каждая задача заканчивается не «слой готов», а «этим можно пользоваться».
Сквозной кривой цикл ценнее идеального слоя. Если срез не приближает момент,
когда основатель читает digest или утверждает Jira-драфт, — это не следующий
срез.

### Правило 2 — usage gate

> Нельзя начинать новую инфраструктурную задачу, пока результат предыдущего
> среза не использован в реальной работе 5 дней подряд. Диагностические отчёты,
> рефакторинг «впрок» и новые защитные слои использованием не считаются.

Чинить то, обо что споткнулись при реальном использовании, — можно всегда.
Строить то, обо что ещё никто не споткнулся, — нельзя.

### Правило 3 — LLM это главный путь

Ценность системы — классификация по смыслу, а не по ключевым словам. Целевой
путь Фазы 1 — включить LLM-триаж для реального daily loop и держать
детерминированный путь как страховку (fallback при невалидном JSON и
conservative-провайдер), как и спроектировано в
`app/services/attention_triage.py`.

Текущее состояние реализации: LLM остаётся default-off/scaffolded и не мутирует
production data. Включение живого LLM-пути требует явного operator approval,
валидируемого strict JSON и eval-покрытия. Выключенный LLM безопасен как
переходное состояние, но не является конечной продуктовой ценностью.

### Правило 4 — evals это позвоночник гибкости

Гибкость системы определяется не архитектурой, а стоимостью проверки
изменений. Золотой набор реальных кейсов в `tests/evals/` гоняется в CI:

- «письмо клиента с вопросом → requires_my_attention»;
- «маркетинговая рассылка с action-словами → hidden/low»;
- «PR назначен мне → high», «PR чужой → important_info/low»;
- «неоднозначный документ → review_optional, не скрыт»;
- «невалидный JSON → retry → conservative fallback».

Каждый случай шума или ошибки триажа из реальной жизни в тот же день становится
eval-кейсом. После этого менять промпты, модели и рендеринг — обычный коммит, а
не прыжок веры. Набор стартует с 30–50 кейсов в Фазе 1 и только растёт.

## Фазы

| Фаза | Содержание | Gate выхода |
|---|---|---|
| **0. Фундамент** | Контракты, guards, raw storage, draft→approval→gated send, CI. **Готово.** | — |
| **1. Daily loop** | Реальный Gmail/Drive backfill → LLM-триаж включён → persisted digest → ручная отправка в Telegram каждое утро → пометки feedback | Digest читается 5 дней подряд; шум < 20%; eval-набор v1 в CI |
| **2. Meetings → Jira** | Транскрипт → summary / decisions / action items → Jira drafts → запись в Jira через approval | 10 реальных задач созданы через approve; решения в decision log |
| **3. GitHub-поток** | События org qtwin-io → digest; PR-review агент по запросу | PR-review комментарии реально используются |
| **4. Роли по запросу** | Агенты ролей (QA/TL/UX/PM) — только те, что кто-то попросил после Фаз 1–3 | Конкретный человек пользуется конкретным агентом неделю |
| **5. Автоматизация** | Scheduler, webhooks, авто-доставка | Месяц стабильного ручного ритма без duplicate-инцидентов |

Внутри фазы — WIP-лимит 1: один срез в работе. «А ещё добавь отчёт» — в бэклог,
не в текущий срез.

## Операционная модель

Рабочий цикл: **чат-ПМ → промпт → Claude Code → результат → чат-ПМ**.

- Промпт задачи — строго по шаблону из `AGENTS.md` (Task Prompts):
  `Goal / Context / Constraints / Done when`, со ссылкой на AGENTS.md вместо
  копирования правил. `Done when` включает `uv run ruff check .` и
  `uv run pytest -q` зелёные. CI также проверяет frozen install, миграции и
  tracked-secret scan; для parity-прогона используйте команды из `README.md`.
- Санитизация — по `AGENTS.md` (Sanitization Scope): секреты — никогда;
  git-метаданные собственного репо — свободно.
- Git-поток — по `AGENTS.md` (Default Git Workflow).

### Еженедельный ритуал (30 минут)

1. Что я реально использовал на этой неделе?
2. Что в digest было шумом? → каждый случай в eval-набор.
3. Какие драфты я отклонил и почему? → корректировка промптов/контекста.
4. Метрики недели (ниже).
5. Один следующий срез на неделю.

### Метрики честности

- Digest read rate: читался ли digest каждый рабочий день.
- Noise rate: доля пунктов digest, помеченных как шум.
- Draft approval rate: доля approve среди Jira/KB-драфтов.
- Hidden-but-important: скрытые пункты, позже оказавшиеся важными
  (самый опасный сигнал — должен быть 0).
- Если digest не читался 3 дня подряд — чиним шум, новые фичи замораживаются.

## Незыблемое ядро

Эти контракты — точки расширения системы; они не меняются без явного решения:

- `NormalizedActivityItem` — единый кросс-source контракт. Новый источник =
  один маппер в этот контракт, всё остальное работает без изменений.
- `AttentionTriageResult` + confidence policy (0.80 / 0.55, low-confidence
  никогда не скрывается молча).
- `evidence_refs` на каждом извлечённом факте; нет evidence — нет факта.
- Цепочка delivery: draft → human approval → intention → gated execution →
  result; duplicate-success защита — финальный страж на отправке.
- Guards (provider / production / scheduler) на каждой живой границе,
  default-deny.
- Raw storage + Postgres — source of truth; Obsidian — export-only;
  Telegram/Slack — интерфейс, не истина.
- Provider abstraction: модель/провайдер LLM сменяемы, evals подтверждают
  эквивалентность.

## Анти-паттерны (что больше не делаем)

- Новые диагностические отчёты/доктора без зафиксированного случая реального
  использования, который они обслуживают.
- Рефакторинг и библиотеки «впрок» до того, как боль возникла трижды.
- Мега-промпты с копией всех правил и запретами на печать git-метаданных.
- Параллельные срезы (WIP > 1).
- Авто-доставка/scheduler раньше Фазы 5.
- Append-only гроссбухи статусов в feature-доках: статусы — в CHANGELOG,
  feature-док описывает текущий контракт. Если старый журнал нужен для
  traceability, держим его под `Historical Implementation Ledger (Archived)`.

## Как развивать систему дальше

- **Новый источник данных** (Slack, Calendar, Figma): маппер →
  `NormalizedActivityItem` + 5–10 eval-кейсов + появление в digest. Всё.
- **Новый агент-роль**: потребитель существующих контрактов
  (`AttentionTriageResult`, digest read model), выход — драфты через
  существующую approval-цепочку.
- **Смена модели/провайдера**: реализация провайдер-интерфейса + зелёный
  eval-набор = можно мержить.
- **Новый канал доставки** (Slack вместо Telegram): новый adapter за теми же
  guard'ами и той же duplicate-защитой по канонич. хэшу.
- Любое из этого — отдельный вертикальный срез со своим usage gate.
